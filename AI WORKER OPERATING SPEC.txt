AI WORKER OPERATING SPEC
For the Decomp Engine project
Version: 0.1

PURPOSE

This document defines how AI workers must operate on the Decomp Engine project.

Its purpose is to prevent:
- circular debugging
- speculative patches
- repeated investigation of already-solved layers
- oversized fixes
- loss of architectural discipline

This spec is mandatory for all AI workers acting on the repository.

==================================================
1. PRIMARY RULE
==================================================

ALWAYS NARROW THE ACTIVE BLOCKER.

Do not work broadly.
Do not “improve” unrelated systems.
Do not reopen old hypotheses unless new evidence directly contradicts them.

At any moment, there should be exactly one active blocker family.

Examples:
- build failure
- unresolved symbol family
- runtime stall at one function boundary
- bad asset ownership for one asset family
- renderer consuming stale buffered state

If multiple problems exist, solve the earliest true blocker first.

==================================================
2. SOURCE OF TRUTH HIERARCHY
==================================================

When evidence conflicts, trust sources in this order:

1. Live runtime trace / debugger / actual execution evidence
2. Build output / linker errors / emitted artifacts
3. Exact source code at ownership point
4. Narrow helper search results
5. Broad helper speculation
6. General theory

This means:
- a runtime trace is stronger than a helper’s “probably”
- a missing post-trace is stronger than a guessed family diagnosis
- a concrete undefined reference is stronger than a suspected ownership issue

==================================================
3. ACTIVE BLOCKER MODEL
==================================================

Every iteration must begin by naming:

ACTIVE BLOCKER:
- one sentence only

Examples:
- stall inside InitIntrHandlers() before post-INTR_VECTOR
- wrong in-memory byte representation for sCopyright_Gfx
- renderer sees DISPCNT/BG0CNT as zero because buffered regs never flush

Every action must be directly tied to that blocker.

Forbidden:
- “while here” cleanup
- side refactors
- unrelated architecture rewrites
- broad speculative fixes

==================================================
4. CLOSED FACTS LEDGER
==================================================

Workers must maintain a CLOSED FACTS ledger.

A CLOSED FACT is a condition already proven by evidence and not to be revisited unless contradicted by new evidence.

Examples:
- GDExtension loads and registers FireRedNode
- ROM loading succeeds
- firered_init() returns success
- InitIntrHandlers completes
- LoadCopyrightGraphics decompresses successfully
- framebuffer texture is non-null

Rules:
- closed facts must not be re-diagnosed
- old hypotheses are not allowed to re-enter the search space unless a new trace or build output contradicts them
- each response should explicitly list the closed facts relevant to the current blocker

==================================================
5. TRACE-FIRST DEBUGGING POLICY
==================================================

For runtime problems, use this order:

1. add trace boundaries
2. rerun
3. identify first missing post-trace
4. only then patch

Never patch first unless the failure is already isolated to one exact operation.

Use pre/post traces at function boundaries:

- FunctionX: enter
- FunctionX: pre-OpA
- FunctionX: post-OpA
- FunctionX: pre-OpB
- FunctionX: post-OpB
- FunctionX: exit

Goal:
identify the first operation that does not return.

Do not:
- add dozens of deep traces at once
- instrument whole subsystems blindly
- trace unrelated layers after a boundary is already known

==================================================
6. PATCH-SECOND POLICY
==================================================

A worker may patch only after one of these is true:

A. the failing ownership point is already isolated
B. the exact helper/function that fails is already isolated
C. a build error explicitly names the missing symbol/file/config issue
D. a representation mismatch is proven byte-for-byte

Patches must be:
- narrow
- local
- reversible
- justified by evidence

Each patch explanation must include:
- why this is the smallest correct move
- what exact blocker it addresses
- what it intentionally does not address

==================================================
7. OWNERSHIP-POINT PATCHING RULE
==================================================

Always patch at the ownership point when possible.

Examples of ownership points:
- function responsible for hardware probing
- exact helper doing bad memory copy
- asset mapping table for one symbol family
- buffered GPU register flush function
- callback dispatcher boundary
- content provider resolution function

Do not patch around the ownership point unless necessary.

Good:
- skip INTR_VECTOR write under PORTABLE inside InitIntrHandlers()

Bad:
- broadly disable startup logic somewhere upstream

==================================================
8. HELPER / SUBAGENT USAGE RULES
==================================================

Helpers are useful for:
- file discovery
- symbol ownership
- call-chain listing
- build-log grouping
- object-family tracing
- identifying likely ownership boundaries

Helpers are NOT the final authority for:
- whether to stub or bypass behavior
- final patch decisions when runtime trace disagrees
- broad architectural changes

Use helpers only for narrow questions.

GOOD helper prompts:
- list first top-level calls inside firered_run_frame
- locate owner of gItems
- trace which object file should define BattleScript_MoveEffectSleep

BAD helper prompts:
- diagnose the whole hang
- figure out what is wrong overall
- decide what subsystem to disable

==================================================
9. WHEN TO STOP USING HELPERS
==================================================

Stop relying on helpers when live evidence already isolates the blocker more precisely than repository search can.

Examples:
- trace stops before post-INTR_VECTOR
- callback dispatch reaches CB2_InitCopyrightScreenAfterBootup and stalls there
- decompressor reports bad-header for a known asset pointer

At that point:
- direct local tracing
- byte-level inspection
- debugger stepping

are stronger than helper guesses.

==================================================
10. NO-CIRCLES RULE
==================================================

If the same layer has been rechecked twice without new contradictory evidence, workers must stop and narrow mechanically.

Examples of prohibited circles:
- revisiting VBlank theory after trace proves stall is earlier
- revisiting extension load after runtime already enters firered_run_frame
- revisiting asset ownership after byte-level truth proves ownership is fixed and representation is wrong

When circling is detected, workers must switch to:
- mechanical boundary tracing
- explicit closed-facts list
- one-function-at-a-time narrowing

==================================================
11. MECHANICAL NARROWING RULE
==================================================

When a blocker is inside one function and the cause is unknown, use binary-style narrowing.

Procedure:
1. list top-level operations in source order
2. add pre/post trace around each top-level operation
3. rerun
4. find first missing post-trace
5. if the failing op is another helper, recurse into that helper only

Do not continue broad reasoning until the first non-returning or faulting helper is known.

This rule is mandatory for:
- runtime stalls
- bad asset decode chains
- first-screen render failures
- early callback hangs

==================================================
12. BYTE-LEVEL TRUTH RULE
==================================================

When the issue concerns asset bytes or decompression, workers must verify:

1. on-disk file bytes
2. runtime bytes at symbol pointer
3. bytes seen by the consuming helper
4. destination pointer validity

Never keep changing ownership or offsets without a byte-level comparison.

If the first bytes in memory do not match the file:
- the problem is representation/storage
not ownership
not decompression
not destination pointer

==================================================
13. RENDER VISIBILITY RULE
==================================================

If decoded assets exist but the displayed frame is blank/wrong, workers must split the problem into:

A. content arrival
- did meaningful data land in VRAM/palette/tilemap buffers?

B. state visibility
- did buffered GPU register/palette state reach the memory/state the renderer consumes?

C. renderer consumption
- is the renderer reading the correct memory/state?

Do not jump from “blank frame” to “assets missing” without evidence.

==================================================
14. BUILD FAILURE RULE
==================================================

For build problems, workers must classify failures into one of these families:

- missing dependency/tool bootstrap
- archive/link ordering
- missing symbol ownership
- generated data/script object omission
- wrong binary flavor / ABI
- wrong path quoting/path conversion
- include/config/macro ownership

Workers must solve:
- the earliest true blocker
- one family at a time

For unresolved symbols:
- identify defining file
- identify expected object/archive member
- explain why it is absent
- patch at ownership boundary

Never add fake stubs for data/script symbols unless explicitly justified and approved.

==================================================
15. PORTABLE PATCH RULE
==================================================

For PORTABLE-only behavior changes:

Allowed:
- skipping real hardware probes
- replacing BIOS helpers with portable equivalents
- redirecting hardware state into software-managed state
- narrowly bypassing impossible hardware operations

Forbidden without strong evidence:
- broad disabling of gameplay systems
- bypassing whole callback chains
- faking whole scenes
- changing game logic because hardware behavior is inconvenient

PORTABLE guards must:
- preserve non-portable behavior unchanged
- be as narrow as possible
- be documented in the worker report

==================================================
16. REPORT FORMAT RULE
==================================================

Every worker response must include:

1. ACTIVE BLOCKER
2. CLOSED FACTS relevant to it
3. EXACT CHANGE MADE
4. WHY THIS IS THE SMALLEST CORRECT MOVE
5. FILES EDITED
6. REBUILD/RERUN RESULT
7. NEXT BLOCKER OR SUCCESS

If tracing only:
- say tracing only
- do not claim a fix

If patching:
- explain why the evidence justified patching

==================================================
17. SUCCESS CONDITION RULE
==================================================

A task is complete only when one of these is true:

A. the active blocker is removed and execution/build moves to a deeper, new blocker
B. the visual output or build artifact is correct enough for the target milestone
C. the exact next blocker is isolated with stronger evidence than before

“Probably fixed” is not completion.

==================================================
18. WORKER ESCALATION RULE
==================================================

Escalate to debugger use when:
- trace narrowing has isolated the problem to one function but not one operation
- repeated trace iterations stop giving new information
- the issue may be crash vs loop vs bad memory and trace cannot distinguish them
- the runtime is now executing deep enough that stepping is cheaper than more instrumentation

Until then, trace-first remains acceptable.

==================================================
19. NEW RUNTIME PORT RULE
==================================================

When adding a new game runtime, workers must follow this order:

1. define runtime shell API binding
2. isolate content/image loading
3. identify hardware-facing dependencies
4. build portable platform replacements
5. verify one frame / one screen
6. verify framebuffer output
7. validate behavior
8. only then expand scope

Do not start with:
- UI
- mod manager integration
- advanced rendering
- platform-specific polish

==================================================
20. ARCHITECTURE PROTECTION RULE
==================================================

Workers must preserve the architecture:

Shell Layer
- UI, app flow, platform packaging

Runtime Layer
- game logic only

Content Layer
- ROM, patches, overrides, saves

Platform Layer
- video/audio/input/timing abstractions

Workers must not:
- push shell logic into runtime
- push game logic into shell
- make mod loading runtime-specific if it belongs in content layer
- create shared “retro soup” abstractions across incompatible game cores

==================================================
21. FIRE RED CURRENT STATE BASELINE
==================================================

Unless contradicted by new evidence, workers should assume:

- root build succeeds
- gdextension build succeeds
- FireRedNode loads and instantiates
- ROM loading works
- firered_init() works
- one frame can execute
- framebuffer handoff works
- current class of issues are likely visual correctness / state visibility / later runtime correctness

This baseline prevents re-diagnosing old phases.

==================================================
22. PREFERRED THINKING STYLE
==================================================

Workers must think in this order:

1. What is the exact active blocker?
2. What has already been proven?
3. What is the narrowest ownership point?
4. What evidence is missing?
5. Do we need trace, helper search, or patch?
6. What is the smallest move that adds new truth?

Never think in this order:
- what whole subsystem might be wrong?
- what can I patch quickly?
- what else can I clean up while here?

==================================================
23. FINAL RULE
==================================================

The worker’s job is not to be impressive.
The worker’s job is to reduce uncertainty.

Every iteration must either:
- remove one blocker
or
- isolate one blocker more precisely

Nothing else counts.

END OF DOCUMENT
